# Adding a page? Update the sitemap.

The `/sitemap` page ([app/sitemap/page.tsx](app/sitemap/page.tsx)) is hand-maintained from
[lib/sitemapData.ts](lib/sitemapData.ts) — it is **not** generated by crawling `app/`. So it does not
update itself.

**Whenever you add a `page.tsx` under `app/`, add a matching entry to `lib/sitemapData.ts` in the same
change.** Whenever you delete or move a page, remove or repoint its entry. Two invariants must hold:

- **No orphans** — every route under `app/` is reachable from the sitemap (except `/sitemap` itself).
- **No dead links** — every `href` in `sitemapData.ts` resolves to a real route. Never use `"#"` here.

An entry is `{ key, href }`, where `key` is a `TranslationKey` and `href` is the route. Reuse an
existing label key if one fits (the nav and footer already define most of them); only add a new key if
none does — and remember a new key must be added to **all four** locale files (`en`, `hi`, `mr`, `ta`),
or the build fails, since `translations` is typed `Record<Language, Record<TranslationKey, string>>`.

Group headings follow one rule: a group with an `href` renders as a teal link, one without renders as a
plain dark section heading.

To check both invariants, compare the routes on disk against the hrefs in the data file:

```bash
node -e '
const {readFileSync,readdirSync,statSync}=require("fs");
const routes=new Set();(function w(d,b=""){for(const e of readdirSync(d)){const p=`${d}/${e}`;
statSync(p).isDirectory()?w(p,`${b}/${e}`):e==="page.tsx"&&routes.add(b||"/")}})("app");
const linked=new Set([...readFileSync("lib/sitemapData.ts","utf8").matchAll(/href:\s*"([^"]+)"/g)].map(m=>m[1]));
const broken=[...linked].filter(h=>!routes.has(h));
const orphans=[...routes].filter(r=>!linked.has(r)&&r!=="/sitemap");
console.log("broken:",broken,"\norphans:",orphans);
process.exit(broken.length||orphans.length?1:0);'
```

It exits non-zero and names the offenders if either invariant is violated.

# Every user-visible string must be translated. No exceptions.

This site ships in four languages — English, Hindi, Marathi, Tamil (`lib/i18n/{en,hi,mr,ta}.ts`).
**Never hardcode user-visible text in a component.** Put it in the catalog and render it through
`t("someKey")`, so it switches with the language.

"User-visible" is wider than the text you can see. It also covers text a screen reader announces or a
browser surfaces, all of which must go through `t()`:

- JSX text nodes
- `alt` on every `<Image>` / `<img>`
- `aria-label`, `title`, `placeholder`, `label`

Adding a string means three things, all in the same change:

1. Add the key to **`en.ts`**.
2. Add it to **`hi.ts`, `mr.ts` and `ta.ts` as well** — this is enforced by the compiler, since
   `translations` is typed `Record<Language, Record<TranslationKey, string>>`; a key missing from any
   locale fails the build.
3. **Actually translate it.** Copying the English value into `hi`/`mr`/`ta` satisfies the compiler but
   is exactly the bug this rule exists to prevent — it compiles, and the page silently stays English.

The only strings that may stay in Latin script are proper nouns and non-linguistic tokens: brand names
(`CIBIL`, `TransUnion`, `RBI`, `NBFC`, `PAN`), the logo tagline ("Part of TransUnion"), the `JAAG₹AN`
wordmark, email addresses, URLs, prices, and input masks like `DD / MM / YYYY`. Everything else gets
translated, including headings, buttons, alt text and aria-labels.

If a component needs `t()` but is a server component, add `"use client"` and
`const { t } = useLanguage();` — that is the established pattern for every `*Content.tsx`.

Two checks. First, find text that never reaches `t()` (scans JSX text plus `alt` / `aria-label` /
`title` / `placeholder`):

```bash
node -e '
const {readFileSync,readdirSync,statSync}=require("fs");const files=[];
(function w(d){for(const e of readdirSync(d)){const p=`${d}/${e}`;
statSync(p).isDirectory()?w(p):/\.tsx$/.test(p)&&files.push(p)}})("components");
const OK=/^(CIBIL|TransUnion|RBI|NBFC|PAN|GST|OTP|ID|MSME|CERSAI|FAQs?|LinkedIn|Facebook|YouTube|Instagram)$/i;
const hits=[];for(const f of files){readFileSync(f,"utf8").split(/\r?\n/).forEach((l,i)=>{
if(/^\s*(\/\/|\*)/.test(l))return;
for(const m of l.matchAll(/>([^<>{}\n]{3,})</g)){const s=m[1].trim();
if(/[A-Za-z]{3,}/.test(s)&&!OK.test(s))hits.push(`${f}:${i+1}  ${s}`)}
for(const m of l.matchAll(/\b(alt|placeholder|title|aria-label)="([^"]{3,})"/g)){const s=m[2].trim();
if(/[A-Za-z]{3,}/.test(s)&&!OK.test(s))hits.push(`${f}:${i+1}  [${m[1]}] ${s}`)}})}
console.log(hits.join("\n")||"none");'
```

Second, find keys whose translation is still byte-identical to the English (i.e. never translated):

```bash
node -e '
const {readFileSync}=require("fs");
const p=l=>{const m=new Map();const re=/^  (\w+):\s*(?:\r?\n\s*)?"([^"]*)"/gm;
let x;const s=readFileSync(`lib/i18n/${l}.ts`,"utf8");while(x=re.exec(s))m.set(x[1],x[2]);return m};
const en=p("en");
for(const l of ["hi","mr","ta"]){const o=p(l);
const same=[...en.keys()].filter(k=>o.get(k)===en.get(k)&&/[A-Za-z]{4,}/.test(en.get(k)||""));
console.log(l+":", same.length?same.join(", "):"ok")}'
```

Review anything either check prints. A hit is only acceptable if it is a proper noun or token from the
list above. As of the last audit both checks are clean: the only values matching English are the brand
strings (`brand`, `homeHeroBrand`, `productsHeadingPrefix`, `aboutUsEyebrow`, `concernsMatterBrand`),
the email addresses, `statQuoteAuthor` and `ccrfCode` — all of which are correct as-is.
